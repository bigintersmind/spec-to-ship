# PRD-vs-implementation review

You're being run by `ralph/review.sh` after `ralph/afk.sh` exits. Your job is to compare what shipped (the diff and commit log already pasted into this prompt) against the PRD's intent and the per-child agent briefs, then post a structured review comment on the PRD ticket.

# Pre-flight

Before doing anything else, read these to learn this repo's conventions:

- `docs/agents/issue-tracker.md` — how to fetch tickets and post comments
- `docs/agents/triage-labels.md` — canonical role → label mapping for this repo

The PRD ref, timestamp, and commit range have been handed to you verbatim by the wrapper script — use them as given. The diff and commit log are already in this prompt; do not re-run `git diff` or `git log`.

# Fetch the PRD

Use the "fetch the relevant ticket" verb from `docs/agents/issue-tracker.md` against the PRD ref to get the PRD body and its comments.

# Discover the children

Locate the comment under the PRD with a `## Children` heading. Its body is a bullet list of child ticket refs (the format is mandated by `/issues`).

If multiple comments with a `## Children` heading exist on the PRD, take the **most recent** by created_at. Older comments are stale leftovers from previous `/issues` runs and must be ignored. If exactly one exists, use it.

If zero `## Children` comments exist, post one short corrective comment on the PRD using the "comment on a ticket" verb:

> *This was generated by AI during review.*
>
> No `## Children` comment found on this PRD — was `/issues` run on this PRD? Skipping review.

…and exit. Do not fall back to scanning the PRD body for ticket references.

# Per-child fetch

For each child ref in the children list, use the "fetch the relevant ticket" verb to read the child. Locate its **Agent Brief** comment (the comment beginning with `> *This was generated by AI during triage.*` and containing a `## Agent Brief` heading). The brief is the durable contract — prefer it over the original ticket body when judging acceptance. If a child has no brief, note that explicitly under Q2 and fall back to the ticket body's acceptance criteria.

# Five-question review

Compare the diff and commit log against the PRD body and the per-child briefs. Answer all five questions in order, no skipping:

1. **PRD coverage** — does the diff deliver each user story in the PRD? Cite specific files / commits when claiming coverage.
2. **Acceptance criteria status** — per child, walk each acceptance criterion and judge it as **satisfied**, **partial**, or **unverifiable** (with reasoning). Note any child missing an agent brief.
3. **Brief vs PRD fidelity** — places the agent brief diverged from the PRD's intent. This is the highest-value question — divergence here is the class of bug per-child review can't see, because each child looks correct against its own brief.
4. **Out-of-scope drift** — anything implemented that the PRD's "Out of scope" list excluded? Anything implemented beyond what the PRD or briefs asked for?
5. **Follow-ups** — concrete follow-up tickets the maintainer should consider. Phrase them as ticket titles, not vague prose.

# Post the review

Use the "comment on a ticket" verb against the PRD ticket with this exact format. Both `<TIMESTAMP>` and `<RANGE>` have been handed to you verbatim from bash — substitute them in but don't change them.

```markdown
> *This was generated by AI during review.*

## Review pass — <TIMESTAMP> (<RANGE>)

### 1. PRD coverage
…

### 2. Acceptance criteria status
…

### 3. Brief vs PRD fidelity
…

### 4. Out-of-scope drift
…

### 5. Follow-ups
…
```

Do not edit, overwrite, or remove any prior `## Review pass` comments. Re-runs append a fresh comment so the PRD's tracker history is a chronological log of what each pass found.
