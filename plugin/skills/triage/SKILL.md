# Triage

Move tickets through the canonical triage state machine. The output is one of:

- A `ready-for-agent` ticket with a durable **agent brief** comment that the AFK loop picks up
- A `ready-for-human` ticket with the same brief structure but flagged for manual handling
- A `needs-info` ticket with specific questions for the reporter
- A closed `wontfix` ticket, with prior rejections logged in `.out-of-scope/` for enhancements

The agent brief is the contract the AFK loop reads — not the original ticket body. The brief is written to survive codebase churn that would invalidate file-path references in the body.

## Disclaimer requirement

Every comment or new artifact created during triage **must** start with:

```
> *This was generated by AI during triage.*
```

This makes AI-generated triage activity scannable for the maintainer when reviewing tracker history later.

## Process

### 1. Load the per-repo conventions

This skill speaks in stable verbs — "publish to the issue tracker", "fetch the relevant ticket", "comment on a ticket", "apply the `<role>` triage label", "close the ticket". The concrete behavior for *this* repo lives in two files:

- `docs/agents/issue-tracker.md` — defines the tracker (GitHub, GitLab, local markdown, or other) and the CLI/file-write conventions for each verb
- `docs/agents/triage-labels.md` — maps the canonical roles (`prd`, `bug`, `enhancement`, `needs-triage`, `needs-info`, `ready-for-agent`, `ready-for-human`, `wontfix`) to the actual label strings this repo uses

Read both files before doing any tracker operations. If either is missing, tell the user to run `/setup-skills` first and stop.

Also read `docs/agents/domain.md` if it exists (and the `CONTEXT.md`/ADRs it points to) so your codebase exploration uses the project's domain vocabulary and respects existing architectural decisions.

## Roles

Two **category** roles, applied alongside a state role:

- `bug` — something is broken
- `enhancement` — new feature or improvement

Five **state** roles, exactly one at a time:

- `needs-triage` — maintainer needs to evaluate
- `needs-info` — waiting on reporter for more information
- `ready-for-agent` — fully specified, AFK-ready
- `ready-for-human` — needs human implementation
- `wontfix` — will not be actioned

A triaged ticket should carry exactly one category role and one state role. If state roles conflict, flag it and ask the maintainer before doing anything else.

State transitions: an unlabeled ticket normally goes to `needs-triage` first; from there it moves to `needs-info`, `ready-for-agent`, `ready-for-human`, or `wontfix`. `needs-info` returns to `needs-triage` once the reporter replies. The maintainer can override at any time — flag transitions that look unusual and ask before proceeding.

## PRDs are not work tickets

Tickets with the `prd` kind label are planning artifacts created by `/prd`. They aren't subject to the state machine above — there's nothing to implement in a PRD, only children to spawn.

If asked to triage a `prd`-kinded ticket, decline and offer the right tool:

> "This is a PRD, not a work ticket — the triage state machine doesn't apply. Want me to run `/issues <ref>` to break it down into child tickets, or check whether all its existing children have been triaged?"

When showing what needs attention, exclude `prd`-kinded tickets — they're not waiting on triage.

## Interpreting the request

The user invokes `/triage` and describes what they want in natural language. Branch on what they're asking for:

- "Show me anything that needs my attention" → see "Show what needs attention"
- "Let's look at #42" / "triage #42" → see "Triage a specific ticket"
- "Move #42 to ready-for-agent" → see "Quick state override"
- "What's ready for agents to pick up?" → list tickets with `ready-for-agent` applied, sorted oldest first

## Show what needs attention

Query the issue tracker (per `docs/agents/issue-tracker.md`) and present three buckets, oldest first:

1. **Unlabeled** — never triaged
2. **`needs-triage`** — evaluation in progress
3. **`needs-info` with reporter activity since the last triage notes** — needs re-evaluation

Show counts and a one-line summary per ticket. Let the maintainer pick.

## Triage a specific ticket

1. **Gather context.** Fetch the relevant ticket. Read its full body and comments. Parse any prior triage notes so you don't re-ask resolved questions. Explore the codebase using the project's domain glossary, respecting ADRs in the area. Read all `.out-of-scope/*.md` files at the repo root and surface any prior rejection that resembles this ticket.

2. **Recommend.** Tell the maintainer your category and state recommendation with reasoning, plus a brief codebase summary relevant to the ticket. Wait for direction before proceeding.

3. **Reproduce (bugs only).** Before any grilling, attempt reproduction: read the reporter's steps, trace the relevant code, run tests or commands. Report what happened — successful repro with code path, failed repro, or insufficient detail (a strong `needs-info` signal). A confirmed repro makes a much stronger agent brief.

4. **Grill (if needed).** If the ticket needs fleshing out before it can become a useful agent brief, run a `/spec` session to walk the design tree and resolve open questions. The grilling output feeds directly into the brief.

5. **Apply the outcome:**
   - **`ready-for-agent`** — comment on the ticket with an agent brief (see [AGENT-BRIEF.md](AGENT-BRIEF.md)), then apply the `ready-for-agent` triage label.
   - **`ready-for-human`** — comment on the ticket with the same brief structure, but note why it can't be delegated (judgment calls, external access, design decisions, manual testing). Apply the `ready-for-human` triage label.
   - **`needs-info`** — comment with the needs-info template below, then apply the `needs-info` triage label.
   - **`wontfix`** (bug) — comment with a polite explanation, then close the ticket and apply the `wontfix` triage label.
   - **`wontfix`** (enhancement) — write to `.out-of-scope/` per [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md), comment linking to the file, then close the ticket and apply the `wontfix` triage label.
   - **`needs-triage`** — apply the role. Optional comment if there's partial progress.

   Apply the matching `bug` or `enhancement` category label as well, if not already set.

   Don't forget the disclaimer prefix on every new comment.

## Quick state override

If the maintainer says "move #42 to ready-for-agent", trust them and apply the role directly. Confirm what you're about to do (label changes, comment, close), then act. Skip grilling.

If moving to `ready-for-agent` without a grilling session, ask whether they want to write an agent brief — without one, the AFK loop has no contract to work from and will refuse to pick the ticket up.

## Resuming a previous session

If prior triage notes exist on the ticket, read them, check whether the reporter has answered any outstanding questions, and present an updated picture before continuing. Don't re-ask resolved questions.

## Needs-info template

```markdown
> *This was generated by AI during triage.*

## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

Capture everything resolved during grilling under "established so far" so the work isn't lost. Questions must be specific and actionable, not "please provide more info".
